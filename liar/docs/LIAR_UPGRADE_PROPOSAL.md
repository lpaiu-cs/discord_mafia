# LIAR_UPGRADE_PROPOSAL.md

## 문서 목적

- 이 문서는 `liar` 게임의 다음 업그레이드 방향을 정리한 제안서다.
- 현재 전제는 `게임 규칙은 유지`하고, `UI/UX`, `연출`, `단어 리소스`, `반복 플레이 유도`를 강화하는 것이다.
- 구현 전 설계 문서로 사용하며, 단계별 마일스톤과 검증 기준을 함께 둔다.

## 현재 판단

`liar` 는 이미 한 판을 완결할 수 있지만, 장기적으로 정말 자주 다시 하게 만들려면 규칙보다 다음 네 가지가 더 중요하다.

1. 진행 화면이 더 직관적이어야 한다.
2. 입력 마찰이 더 낮아야 한다.
3. 종료 순간의 폭발감과 리매치 유도가 더 강해야 한다.
4. 단어 리소스의 깊이와 공정성이 더 높아야 한다.

## 현재 상태 요약

기준 코드:

- 엔진: `liar/src/engine/game.ts`
- Discord 진행: `liar/src/discord/service.ts`
- 카테고리 로더: `liar/src/content/categories.ts`
- 현재 리소스: `liar/resource/categories.json`

현재 특징:

- 로비, 설명, 토론, 투표, 추리, 종료까지 구현 완료
- `/제시어`, `/liar stats`, 버튼, `!투표`, `!스킵` 사용
- 상태 메시지와 타이머 경고 존재
- 카테고리는 `food`, `animal`, `job`, `place`, `object` 수준의 얕은 기본 팩
- `modeB` 는 현재 가능한 카테고리 조합 전체에서 무작위로 선택

## 문제 정의

### 1. 신규/모바일 유저 입력 마찰

- 투표가 `!투표 @대상` 중심이라 모바일에서 번거롭다.
- `!스킵` 도 규칙을 아는 사람이 주도해야 한다.
- `/제시어` 는 기능은 충분하지만 UX는 아직 건조하다.

### 2. 상태 메시지의 행동 유도력이 약함

- 지금도 정보는 들어 있지만, 단계별 핵심 CTA가 충분히 강하지 않다.
- `지금 누구 차례인지`, `뭘 해야 하는지`, `얼마나 남았는지`가 더 즉각적으로 보여야 한다.

### 3. 결과 공개와 리매치 유도 약함

- 결과는 기능적으로 공개되지만, 감정이 터지는 마무리 카드가 약하다.
- 종료 직후 `한 판 더` 흐름이 제품의 기본 종료 경험으로 고정되어 있지 않다.

### 4. 단어 리소스 깊이 부족

- 현재 카테고리 수와 단어 수가 반복 플레이에 부족하다.
- 난이도/민감도/톤/조합 품질을 데이터로 관리할 수 없다.
- `modeB` 에서 재미 있는 조합과 재미 없는 조합을 구분할 수 없다.

## 업그레이드 원칙

### 원칙 1. 규칙은 건드리지 않는다

- 승패 구조, 기본 흐름, `modeA`, `modeB` 구조는 유지한다.
- 업그레이드는 `진행 경험`, `입력`, `연출`, `리소스 품질`에 집중한다.

### 원칙 2. 한 단계씩 개선한다

- 규칙 해석, UI 개편, 리소스 마이그레이션, 전적 UX를 한 번에 섞지 않는다.
- 작은 마일스톤으로 나눠 테스트와 검증 기준을 붙인다.

### 원칙 3. 반복 플레이를 최우선으로 둔다

- 기능의 기준은 `있으면 좋아 보이는가`가 아니라 `한 판 더 하게 만드는가`다.

### 원칙 4. Discord 현실에 맞춘다

- 장기적으로 `Message Content Intent` 의존을 줄인다.
- 버튼, select, slash command 등 Discord 기본 상호작용을 더 활용한다.

## 업그레이드 축

## A. 상태 UI/진행 UX

목표:

- 상태 메시지 하나만 봐도 현재 행동이 이해된다.

개선안:

- 단계별 전용 문구 강화
  - `설명 단계`: `지금 말할 사람`, `한 줄 설명`, `남은 시간`
  - `토론 단계`: `자유 토론`, `과반 !스킵 가능`
  - `투표 단계`: `지금은 투표만 가능`, `현재 투표 수`
  - `추리 단계`: `지목된 라이어만 정답 입력`
- 참가자 필드 가독성 개선
  - 현재 차례, 방장, 완료 여부를 구분
- 상태 색상/아이콘 강화
  - 로비, 진행, 위기, 종료 구간을 시각적으로 분리
- 모바일 기준 줄 수 최적화

## B. 입력 UX 개선

목표:

- 명령을 외우지 않아도 플레이가 된다.

개선안:

- 투표 입력을 단계적으로 버튼/select 기반으로 옮기는 설계 시작
- `!투표`, `!스킵` 은 당장 제거하지 않고 병행 지원
- `/제시어` 응답을 단순 텍스트가 아니라 `역할 카드`처럼 정리
  - 당신의 상태
  - 카테고리
  - 제시어
  - 지금 해야 할 행동

## C. 종료 경험/리매치 루프

목표:

- 결과 공개가 `정보 전달`을 넘어서 `감정 폭발 + 리매치 유도`가 된다.

개선안:

- 종료 결과 카드
  - 승리 팀
  - 라이어 공개
  - 정답 제시어
  - `modeB` 오답 제시어
  - 종료 사유
- 투표 요약
  - 누가 누구를 찍었는지
- 하이라이트 문구
  - 예: `끝까지 들키지 않은 라이어`, `가장 먼저 지목된 플레이어`
- `리매치` 버튼 추가
  - 종료 화면의 첫 CTA 로 설계

## D. 단어 리소스/카테고리 품질

목표:

- 오래 해도 질리지 않고, 억까가 적고, `modeB` 도 재미 편차가 적다.

개선안:

- 카테고리 수 대폭 확장
- 단어 메타데이터 추가
  - 난이도
  - 별칭
  - 태그
  - 민감도
  - 모드 허용 여부
- `modeB` 조합 테이블 도입
  - 가능한 조합을 명시적으로 관리
  - 가중치 기반 추천 가능
- 운영 피드백 기반 품질 관리
  - 너무 쉬운 단어
  - 너무 어려운 단어
  - 불쾌감/정치/민감성 높은 단어

## E. 전적/콘텐츠 UX

목표:

- `/liar stats` 가 텍스트 벽이 아니라 다시 하게 만드는 기록이 된다.

개선안:

- 전적 메시지 구조 정리
- 최근 5판 가독성 개선
- 카테고리/역할별 전적 강조
- 추후 개인/길드 하이라이트 카드와 연결

## 마일스톤 제안

### M1-A. 설계 고정

범위:

- 이 제안 문서 작성
- `단어 리소스 v2` 스키마 문서 작성
- 샘플 카테고리 구조 작성

종료 기준:

- 구현 전에 참조할 문서가 존재

### M1-B. 상태 메시지/결과 카드

범위:

- `liar/src/discord/service.ts`
- 관련 Discord 서비스 테스트

검증:

- 각 단계 상태 메시지에 핵심 CTA가 보인다
- 결과 카드가 기존 결과 정보 손실 없이 출력된다

### M1-C. 리매치 루프

범위:

- 종료 상태 버튼/로비 재생성 흐름
- 상태 초기화/오디오 초기화/전적 기록 순서 검증

검증:

- 종료 직후 같은 서버에서 바로 새 판을 열 수 있다
- 전적 저장이 중복되지 않는다

### M1-D. 리소스 v2 로더 도입

범위:

- `liar/src/content/categories.ts`
- `liar/resource/categories.json` 또는 신규 v2 파일 어댑터
- 관련 테스트

검증:

- 기존 v1 과 호환되거나 마이그레이션 경로가 분명하다
- 카테고리, 단어, `modeB` 조합 검증 로직이 생긴다

### M1-E. 콘텐츠 확장

범위:

- 카테고리 추가
- 단어 품질 관리
- 길드 override 전략 정리

검증:

- 기본 팩만으로도 반복 플레이 피로가 줄어든다
