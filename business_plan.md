# business_plan.md

## 문서 목적

- 이 문서는 `discord_mafia`를 단순 취미 프로젝트가 아니라 장기 운영 가능한 제품으로 전환하기 위한 사업 계획 초안이다.
- 현재 저장소의 현실을 반영한다.
  - Discord 로비 + 웹 대시보드 구조
  - 시즌4 일반·클래식 일부 복제 범위
  - 현재는 단일 노드 중심 운영
- 특히 향후 `플레이 기록`, `승패`, `판수`, `시즌`, `랭킹`, `과금`, `운영 지표`를 저장할 수 있는 DB 확장 방향을 중심축으로 둔다.

## 한 줄 전략

`마피아42 복제 봇`에서 끝내지 않고, `Discord 기반 소셜 추리게임 운영 플랫폼`으로 제품 정의를 넓히는 것이 중장기적으로 맞다.

직접 돈을 받기 시작하는 시점부터는 다음 세 가지가 동시에 필요하다.

- 법적 리스크를 줄일 수 있는 제품 포지셔닝과 브랜딩
- 플레이 기록과 과금 상태를 안정적으로 다루는 영속 DB
- 운영자가 돈을 낼 이유가 분명한 서버용 기능과 플레이어가 돈을 낼 이유가 분명한 개인 기능

## 현재 위치 진단

### 강점

- Discord 안에서 모집이 쉽다.
- 웹 대시보드로 실제 플레이 경험을 Discord 채팅형 봇보다 훨씬 정교하게 만들 수 있다.
- 룰 문서와 엔진이 분리되어 있어 향후 시즌/모드 확장이 가능하다.
- 플레이어별 비공개 정보, 비밀 채팅, 단계별 행동 UI가 이미 제품 형태를 갖추고 있다.

### 약점

- 현재 제품 정의가 `마피아42 시즌4 복제`에 매우 가깝다.
- 플레이 기록, 계정 단위 프로필, 시즌, 랭킹, 결제, 고객지원 같은 사업 핵심 계층이 아직 없다.
- 단일 노드 운영 전제라 확장성, 복구, 데이터 분석 체계가 초기 단계다.
- 운영/법무/개인정보/결제 정책 문서가 아직 없다.

### 가장 큰 사업 리스크

- `IP/브랜딩 리스크`
  - 현재 설명과 구현이 특정 원작의 시즌 룰 복제에 가깝다.
  - 무료 팬 프로젝트와 유료 상용 서비스는 리스크 수준이 다르다.
- `신뢰 리스크`
  - 승패 기록, 시즌 점수, 결제 상품이 들어가면 데이터 손실이나 판정 오류가 곧 사업 리스크가 된다.
- `재방문 리스크`
  - 단순 매치 1회 경험만으로는 장기 유지가 어렵다.
  - 누적 프로필, 시즌 보상, 길드 리더보드 같은 이유가 있어야 남는다.

## 제품 비전

### 단기 비전

- Discord 서버 운영자가 손쉽게 소셜 추리게임을 열고 관리할 수 있는 안정적 도구가 된다.

### 중기 비전

- 플레이어마다 누적 전적과 시즌 성과가 쌓이는 `정체성 있는 서비스`가 된다.

### 장기 비전

- 단일 복제 규칙 제품이 아니라, 여러 룰셋과 커뮤니티 기능을 가진 `소셜 추리게임 플랫폼`이 된다.

## 제품 포지셔닝 전환

상용화 직전에는 제품 설명을 아래 방향으로 바꿔야 한다.

- 현재 표현: `마피아42 시즌4 복제 봇`
- 목표 표현: `Discord 기반 소셜 추리게임 플랫폼`, 또는 `마피아/늑대인간 계열 커뮤니티 게임 운영 서비스`

### 이유

- 상용화 단계에서 가장 먼저 부딪히는 건 기술보다 브랜드와 권리 이슈다.
- 특정 게임명과 시즌 규칙 복제를 전면에 내세우면 사업 리스크가 커진다.

### 권장 방향

- 현재 시즌4 밸런스 룰은 `legacy ruleset` 또는 `classic social deduction ruleset` 식으로 내부 호환 모드화
- 외부 브랜딩은 독자 명칭 사용
- 유료화 전에 법률 검토 진행

## DB 확장 전략

## 왜 DB가 필요한가

다음 기능이 들어가는 순간 파일 기반 저장만으로는 부족해진다.

- 개인 승패 기록
- 누적 판수
- 역할별 통계
- 시즌/랭킹/MMR
- 길드별 순위
- 결제 상태
- 운영 로그
- 부정행위 조사

## 권장 DB 방향

초기 상용화 기준 권장안은 `PostgreSQL` 단일 메인 DB다.

이유는 다음과 같다.

- 관계형 데이터가 많다.
- 승패 정산은 트랜잭션 일관성이 중요하다.
- 길드/유저/매치/참가자/시즌/구독 관계가 복잡하다.
- 향후 분석용 read replica, warehouse 연동도 쉽다.

## DB 단계별 로드맵

### Phase 0: 현재

- 게임 진행 상태는 메모리 기반 런타임 상태
- 목적
  - 빠른 개발
  - 짧은 판 단위 운영 단순화
  - 게임 종료 후 비즈니스 데이터만 별도 저장

### Phase 1: 운영 DB 도입

- 목표
  - 파일 저장을 운영 백업 용도로 줄이고, 서비스의 기준 저장소를 Postgres로 전환
- 범위
  - 유저
  - 길드
  - 매치
  - 참가자 결과
  - 세션
  - 구독/권한

### Phase 2: 전적/시즌 계층 도입

- 목표
  - 유저가 다시 돌아오는 이유 만들기
- 범위
  - 시즌
  - MMR 또는 내부 평점
  - 역할별 통계
  - 길드 리더보드
  - 업적

### Phase 3: 분석/운영/추천 계층 도입

- 목표
  - 수익성과 운영 효율 개선
- 범위
  - 퍼널 분석
  - 리텐션 분석
  - 과금 전환 분석
  - 서버 활성도 분석
  - 이벤트 효과 측정

## 데이터 원칙

- `최종 결과`와 `상세 이벤트`를 분리 저장한다.
- 랭킹 산출에 필요한 데이터는 append-only 성격으로 남긴다.
- 실시간 게임 진행 중 민감한 정보는 최소 권한으로만 접근한다.
- 게임 종료 전 비공개 정보는 분석 계층으로 즉시 흘리지 않는다.

### 3차 기능

- 길드 내 순위

- 멀티 노드 자동 스케일
- 복잡한 실시간 추천
- 과한 재화 시스템
- 불필요한 소셜 피드

## 성공 조건

이 프로젝트가 사업으로 성립하려면 다음을 만족해야 한다.

- 서버 운영자가 `이 봇이 없으면 게임 운영이 불편하다`고 느낀다.
- 플레이어가 `내 전적이 남기 때문에 다시 들어온다`고 느낀다.
- 돈을 내는 기능이 `게임 승패에 직접 영향을 주지 않는다`.
- 데이터 손실과 판정 오류가 사업을 흔들 수준으로 발생하지 않는다.
- 제품이 특정 원작의 단순 복제로만 인식되지 않는다.
